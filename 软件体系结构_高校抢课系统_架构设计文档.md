# 《软件体系结构》课程项目 —— 高校抢课系统 架构设计文档

> 本文档对应课程实验 T1~T4（需求分析、需求分析方法、架构设计决策、ATAM 评估），
> 并附 T5/T6 实现与测试计划，作为后续开发与实验报告撰写的依据。

---

## 0. 项目概述

| 项 | 内容 |
|---|---|
| 题目 | 高校抢课系统（高并发 / 性能权衡型） |
| 技术栈 | Java + Spring Boot（前端 Vue3 + Vite） |
| 架构形态 | 模块化单体 + Redis + Kafka + 限流 |
| 团队/工期 | 2–3 人 / 2–3 周 |
| 核心理念 | 不追求真实海量流量，而是把高并发架构"招式"用对、能压测出来、能在答辩中讲清权衡 |
| 北极星指标 | **任意并发下，某课程已选人数永不超过容量（绝不超卖）** |

**选题理由**：这门课考核重点是"能否把体系结构知识讲出门道"。高校抢课系统在高并发下让
**性能 / 可用性 / 正确性** 三个质量属性产生真实张力 —— ATAM 的敏感点、权衡点、风险点全都有得写；
同时正确性指标"不超卖"极易自动化验证，需求人人都懂，复杂度与可演示性兼备。

## 1. T1 需求分析（功能性 + 非功能性需求）

### 1.1 角色
- **学生**：抢课系统的高并发请求来源。
- **教务管理员**：发布课程、管理抢课批次（开放/关闭时间窗口）、查看统计。
- **系统**：在抢课开放窗口内承接并发选课，保证正确与可用。

### 1.2 功能性需求（用例）
| 用例 | 角色 | 说明 | 是否高并发核心 |
|---|---|---|---|
| UC1 登录认证 | 学生/管理员 | 账号登录，签发 JWT | 否 |
| UC2 浏览/查询课程 | 学生 | 课程列表、详情、剩余名额 | 读高频 |
| **UC3 抢课（选课）** | 学生 | 在开放窗口内对有容量限制的课程选课 | **是（核心）** |
| UC4 退课 | 学生 | 退选并释放名额 | 中 |
| UC5 查看个人课表 | 学生 | 已选课程列表 | 否 |
| UC6 发布/管理课程 | 管理员 | 设置课程容量、上课时间、所属批次 | 否 |
| UC7 管理抢课批次 | 管理员 | 设定抢课开放/关闭时间窗口 | 否 |
| UC8 选课统计 | 管理员 | 各课程报名人数/满员情况 | 否 |

### 1.3 非功能性需求（质量属性）
- **性能 Performance**：抢课开放瞬间，目标并发下 95% 选课请求响应 < 1s，系统不崩溃。
- **可用性 Availability**：抢课期间核心选课服务可用；某依赖（DB/Redis/Kafka）抖动时**降级**而非整体宕机。
- **正确性/一致性 Correctness**：任意并发下不超容量（不超卖）；同一学生对同一课程不重复选课。
- **安全性 Security**：JWT 身份认证；只能操作本人选课；接口限流防刷。
- **可修改性 Modifiability**：库存扣减策略可替换（Redis 原子 ↔ DB 乐观锁）；新增冲突规则不影响核心链路。
- **可测试性 Testability**：不超卖、限流、降级均可自动化验证。

## 3. T3 架构设计决策

### 3.1 总体架构风格
**分层架构 (Layered) + 事件驱动 (Event-Driven，基于 Kafka 异步削峰) + 客户端-服务器 (B/S)**，
工程上打包为**模块化单体 (Modular Monolith)**。

```
┌─────────────────────────────────────────────┐
│ 表现层  Web SPA (Vue3) + REST Controller      │
├─────────────────────────────────────────────┤
│ 应用层  用例编排 / JWT鉴权(Filter) / 限流      │
├─────────────────────────────────────────────┤
│ 领域层  模块: user | course | enrollment      │
│         | batch | common(策略/降级)            │
├─────────────────────────────────────────────┤
│ 基础设施层  MySQL | Redis | Kafka              │
└─────────────────────────────────────────────┘
```
模块包结构：`user / course / enrollment / batch / common`，模块间通过 Service 接口交互，边界清晰。

### 3.2 关键架构决策（Design Decisions）
- **DD1 缓存预热 + Redis 原子扣减库存**：抢课前把各课程容量加载进 Redis；选课请求用 **Lua 脚本**
  原子判断并扣减剩余名额。→ 同时满足"不超卖"与"高性能"（避开每次 DB 行锁）。
- **DD2 Kafka 异步落库（削峰填谷）**：Redis 扣减成功后发"选课成功"事件到 **Kafka 主题 `enroll-events`**，
  消费者组按稳定速率异步写 MySQL 选课记录。Kafka 作为"缓冲水库"把瞬时写峰值平滑为稳定写入。
  **主题按 courseId 分区**（同一课程事件落同一分区，保证分区内有序），消费者组并行消费实现横向扩展。
- **DD2' 为什么选 Kafka 而非 RabbitMQ**：抢课重"扛瞬时洪峰吞吐"，Kafka 的日志型高吞吐 + 分区并行
  + 可重放最贴合削峰场景；RabbitMQ 延迟更低、路由更灵活但吞吐与重放能力弱于 Kafka。代价：Kafka
  默认 **at-least-once** 投递可能重复 → 依赖幂等消费（见 DD4）。
- **DD3 应用层限流（令牌桶 / Redis 计数）**：对选课接口按"用户 + 全局"限流，超额直接拒绝。→ 保护后端、防刷。
- **DD4 防重复选课（双保险）**：Redis Set 记录"用户-课程"已选集合（快速路径）+ MySQL 唯一索引
  `(student_id, course_id)`（兜底，也实现 Kafka 幂等消费）。
- **DD5 库存扣减策略可插拔**：接口 `InventoryDeductionStrategy`，实现 A=Redis 原子、实现 B=DB 乐观锁(version)。
  → 满足可修改性，并作为 ATAM 对比素材。
- **DD6 降级策略**：Kafka 不可用→回退同步写库；Redis 不可用→回退 DB 乐观锁。有损性能但保证可用与正确。
- **DD7 最终一致性 + 对账**：Redis(快速路径) 与 MySQL(持久记录) 通过 Kafka 达成最终一致；定时对账任务校正偏差。

### 3.3 核心抢课链路（数据流）
1. 学生点击抢课 → 前端 `POST /api/enroll {courseId}`
2. **应用层 Filter**：JWT 鉴权 → 限流（Redis 计数/令牌桶），超额直接返回"当前人数过多"
3. **选课服务**：
   - a. 查 Redis 已选集合（防重）→ 已选则返回"你已选过"
   - b. 执行 Lua 脚本：原子判断剩余名额并扣减
     - 名额不足 → 立即返回"已抢完"
     - 扣减成功 → 写入 Redis 已选集合 → 发 Kafka "选课成功"事件（key=courseId）→ 立即返回"抢课成功(处理中)"
4. **Kafka 消费者组**：异步写 MySQL `enrollment` 记录（唯一索引幂等）+ 更新已选计数
5. **失败补偿**：写库失败 → 回补 Redis 名额 + 重试/告警

退课链路：删除 enrollment → 回补 Redis 名额与已选集合（同样幂等）。

### 3.4 数据模型（MySQL）
- `student(id, no, name, password_hash)`
- `course(id, name, capacity, teacher, time_slot, batch_id)`
- `enrollment(id, student_id, course_id, status, created_at)` —— **唯一索引 (student_id, course_id)**
- `enroll_batch(id, name, open_at, close_at)`

## 4. T4 ATAM 体系结构评估

### 4.1 效用树（Utility Tree）
顶层质量属性：性能 / 可用性 / 正确性 / 可修改性 / 安全性 —— 每个挂 1~2 个 QAS 叶子节点，
并标注（重要度, 实现难度）。

### 4.2 敏感点 Sensitivity Points
- **SP1**：库存扣减放 Redis 还是 DB —— 直接同时决定性能与正确性。
- **SP2**：Kafka 可靠性配置（`acks=all` / offset 提交时机 / 幂等消费）—— 决定正确性与可用性。
- **SP3**：限流阈值 —— 决定性能保护与用户体验（误拒率）。

### 4.3 权衡点 Tradeoff Points
- **TP1**：Redis 原子扣减 + Kafka 异步落库 → **性能↑ 可用性↑，但一致性退化为最终一致**。← 最核心权衡点。
- **TP1'**：Kafka vs RabbitMQ → 选 Kafka 得**高吞吐 + 可重放**，代价是**延迟略高 + at-least-once 可能重复**（靠幂等抵消）。
- **TP2**：同步强一致写库（DB 行锁/乐观锁）→ **正确性强，但高并发吞吐↓、响应时间↑**。
- **TP3**：限流 → **保护系统(可用性↑)，但牺牲部分用户成功率（可能误拒合法用户）**。

### 4.4 有风险决策 Risks
- **R1**：Redis 单点故障 → 权威库存丢失（若无持久化/主从）。
- **R2**：Kafka 消息丢失/重复（at-least-once）→ 漏写或重复选课（需幂等消费）。
- **R3**：缓存与 DB 对账不及时 → 长时间不一致。

### 4.5 无风险决策 Non-Risks
- **NR1**：本规模/团队下采用**模块化单体而非微服务** —— 复杂度低、足以支撑需求，是合理决策。
- **NR2**：DB 唯一索引兜底防重复选课 —— 成熟可靠。
- **NR3**：JWT 无状态鉴权 —— 标准方案。

### 4.6 缓解措施
Redis 主从 + 持久化（缓解 R1）；Kafka `acks=all` + 消费成功后再提交 offset + 基于唯一选课ID/DB
唯一索引的幂等消费（缓解 R2）；定时对账任务（缓解 R3）。

## 5. T5 实现计划

**后端依赖**：spring-boot-starter-web、spring-data-jpa(MySQL)、spring-data-redis(Lettuce)、
**spring-kafka**、JWT(jjwt)、限流(Bucket4j 或自写 Redis Lua 令牌桶)。
**前端**：Vue3 + Vite SPA（登录 / 课程列表 / 抢课按钮 / 我的课表 / 管理员发布课程）。
**中间件**：MySQL、Redis、**Kafka（KRaft 模式，免 ZooKeeper）**，用 docker-compose 一键起。

**实现优先级（保证做得完）**：
1. 基础骨架：分层包结构、实体、登录/JWT、课程 CRUD（UC1/UC2/UC6）。
2. **核心抢课链路**（DD1+DD4）：Redis 缓存预热 + Lua 原子扣减 + 防重 + 同步写库版本先跑通。
3. 异步化（DD2）：接入 Kafka（主题 `enroll-events`，按 courseId 分区）异步落库 + 幂等消费 + 失败回补。
4. 限流（DD3）+ 降级（DD6）+ 策略可插拔（DD5，第二实现 DB 乐观锁）。
5. 退课、个人课表、统计、对账任务（UC4/UC5/UC8/DD7）。

> 时间紧时，4、5 中的降级与对账可"已设计 + 部分实现 + 报告中说明"，优先保证核心链路与压测可演示。

**关键文件/模块**：
- `enrollment/EnrollmentController` · `enrollment/EnrollmentService`
- `enrollment/strategy/InventoryDeductionStrategy`（+ `RedisAtomicStrategy` / `DbOptimisticLockStrategy`）
- `enrollment/mq/EnrollEventProducer`（KafkaTemplate）· `EnrollEventConsumer`（@KafkaListener，消费者组）
- `common/filter/JwtAuthFilter` · `common/ratelimit/RateLimiter`
- `resources/lua/deduct_stock.lua`

## 6. T6 测试计划（功能 + 非功能）

### 6.1 功能测试（JUnit + 接口测试）
正常抢课成功 / 名额抢完被拒 / 重复抢课被拒 / 窗口外被拒 / 退课释放名额 / 管理员发布课程 / 鉴权失败被拒。

### 6.2 非功能测试
- **不超卖验证（核心，QAS-2）**：对容量=K 的课程发起 ≫K 并发请求，断言最终 `enrollment` 数 == K、超卖=0。
- **性能/并发（QAS-1）**：JMeter/Gatling 模拟 N 并发抢课，测 QPS、P95/P99 响应时间、错误率，
  对比"同步写库 vs Redis+Kafka"两种策略（呼应 TP1/TP2）。
- **可用性/降级（QAS-3）**：手动停掉 Redis 或 Kafka，验证降级路径仍能正确选课（有损性能）。
- **限流（QAS-5）**：超阈值压测，验证被拒比例符合配置。

## 7. 端到端验证步骤
1. `docker-compose up`（MySQL/Redis/Kafka）→ 启动 Spring Boot → 启动前端。
2. 管理员发布一门容量=50 的课程并开放批次。
3. 用 JMeter 对 `/api/enroll` 发起 1000 并发，检查：
   - 返回成功的请求数 == 50，其余为"已抢完/限流"；
   - `select count(*) from enrollment where course_id=?` == 50（**不超卖**）；
   - Redis 剩余名额 == 0，与 DB 对账一致；
   - 记录 P95 响应时间与 QPS 截图用于报告。
4. 停 Redis 或 Kafka 重复步骤 3，验证降级路径仍不超卖（性能下降可接受）。
