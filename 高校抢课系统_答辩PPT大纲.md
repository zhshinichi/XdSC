# 四、架构评估 ATAM（T4）（P22–P26）

- 【内容要点】一句话讲 ATAM：用效用树梳理质量属性，找出敏感点 / 权衡点 / 风险 / 无风险决策。
- 【版式设计】扉页 + 底部 ATAM 四要素图标。

### P23 · 效用树（Utility Tree）
- 【内容要点】根→5 质量属性→QAS 叶子，每叶标（重要度,难度）。
- 【版式设计】整页放图。
- 【使用素材】**`diagrams/07_效用树.png`**；`ATAM效用树.md`。

### P24 · 敏感点与权衡点
- 【内容要点】SP1 库存放 Redis/DB、SP2 Kafka 可靠性配置、SP3 限流阈值；**核心权衡 TP1**：Redis 原子+Kafka 异步 → 性能↑可用↑ **但一致性退化为最终一致**；TP2 同步强一致写库吞吐↓；TP3 限流保护 vs 误拒。
- 【版式设计】上半敏感点、下半权衡点；TP1 用红框强调。
- 【使用素材】`架构设计文档.md` §4.2、§4.3。

### P25 · 有风险 / 无风险决策 + 缓解
- 【内容要点】风险 R1 Redis 单点、R2 Kafka 重复/丢失、R3 对账不及时；无风险 NR1 模块化单体（非微服务合理）、NR2 唯一索引、NR3 JWT；缓解：Redis 主从+持久化 / acks=all+幂等消费 / 定时对账。
- 【版式设计】左红（风险）右绿（无风险）双栏 + 底部缓解措施条。
- 【使用素材】`架构设计文档.md` §4.4–4.6。

### P26 · ATAM 评估结论
- 【内容要点】结论：当前架构在"性能/可用性"上达标，代价是"最终一致"——这是可接受且已用对账/幂等缓解的有意识取舍；模块化单体匹配本规模。
- 【版式设计】一句话结论 + 三条要点。
- 【使用素材】无（提炼自上面）。

### P34 · 落库策略对比：异步削峰的真正价值（ATAM TP1）
- 【内容要点】**诚实且有深度的一页**。容量 2000 全量并发下对比 async（Kafka）vs sync（写库）：
  - **DB 空载时**：两者吞吐相近，本地 sync 甚至略高（异步 324 vs 同步 584 rps）——因为瓶颈不在 DB，Kafka 反成额外开销。
  - **DB 成为瓶颈时**（实验给单次写入注入 **40ms** 延迟，模拟远程/高负载库）：**异步 1054 rps vs 同步 156 rps ≈ 6.8×**，平均受理延迟 1011ms vs 6557ms。机理：同步请求线程全卡在慢写上（10 连接×40ms 封顶）；异步只做 Redis+发 Kafka 即返回，慢写甩给后台消费者补齐。
  - 结论：异步削峰换的是**可用性 + 削峰 + 写扩展**，其性能优势在「DB 是瓶颈」时才显现——**架构战术的价值取决于真实瓶颈**，这正是 ATAM 权衡点 **TP1**。
- 【版式设计】左"DB 空载"对比图（图08），右"DB 瓶颈 6.8×"对比图（图12），下方一句话结论；用箭头点出"瓶颈在哪，决定异步是否更优"。
- 【使用素材】**`diagrams/08_压测吞吐对比.png`**（DB 空载）+ **`diagrams/12_压测_DB瓶颈下异步优势.png`**（DB 瓶颈，6.8×）；数字取 `高并发压测分析报告.md` §5.3。
